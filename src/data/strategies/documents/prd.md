# PRD : cadrer ce que le produit doit faire avant d'écrire du code

Le PRD sert à répondre à une question simple : qu'est-ce qu'on construit, pour qui, et pourquoi maintenant ?

## Ce qu'un bon PRD contient

- le problème utilisateur ;
- le contexte ;
- l'hypothèse ;
- les critères de succès ;
- les non-objectifs ;
- le périmètre fonctionnel ;
- les points ouverts.

## Ce qu'il ne doit pas devenir

- une spécification technique détaillée ;
- une liste de features sans logique ;
- un texte flou qui mélange vision, produit et implémentation.

## Le vrai intérêt du PRD

Il oblige à clarifier :

- quel comportement on veut changer ;
- pour quel utilisateur ;
- avec quel résultat attendu ;
- sur quoi on accepte de ne pas partir.

Quand ces points sont nets, les discussions produit deviennent beaucoup plus propres.
