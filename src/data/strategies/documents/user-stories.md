# User stories : exprimer un besoin utilisateur sans jargon produit

Les user stories servent à reformuler un besoin du point de vue de l'utilisateur. Leur intérêt n'est pas esthétique. Elles empêchent de parler uniquement en termes de fonctionnalités.

## Le format utile

> En tant que [utilisateur], je veux [action] afin de [bénéfice].

Le point important n'est pas la formule en elle-même. C'est le fait qu'elle t'oblige à nommer le besoin et le bénéfice.

## Une bonne user story doit être

- compréhensible ;
- liée à un usage réel ;
- assez petite pour être traitée ;
- accompagnée de critères d'acceptation.

## Les erreurs classiques

- décrire une solution au lieu d'un besoin ;
- écrire des stories trop grosses ;
- oublier les critères de validation ;
- produire des dizaines de stories sans priorité.

Les user stories sont utiles quand elles clarifient le travail. Si elles deviennent un rituel vide, elles finissent au même endroit que beaucoup de pseudo-process produit : dans un backlog que plus personne ne lit.
